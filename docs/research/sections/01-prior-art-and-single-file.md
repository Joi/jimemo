# Prior art and single-file HTML techniques

Research scope: existing tools that generate self-contained single-file HTML documents/reports, and concrete techniques for embedding all assets (CSS, images, fonts, JS) so a rendered jimemo page has zero remote fetches at view time. All content below is analysis of publicly documented facts; no code was copied from any source into this repository.

| [Pico CSS](https://github.com/picocss/pico) | picocss org | `MIT` | Classless/semantic CSS framework, <10KB compressed, with light/dark themes that switch automatically via `prefers-color-scheme` and no JS required | Direct model for jimemo's theming: ship a small set of CSS custom properties and let semantic HTML (`header`, `main`, `table`) look good with zero authored classes |
| [Tufte CSS](https://github.com/edwardtufte/tufte-css) | Dave Liepmann / Edward Tufte project | `MIT` | CSS-only (deliberately no JS) framework replicating Edward Tufte's handout typography: sidenotes, margin notes, tight figure/caption integration | Directly relevant "ideas worth stealing" for a report-style output — see credited section below |

## Single-file techniques

- **Data-URI size math and overhead.** Base64 encoding inflates binary payloads by roughly 33% (3 bytes become 4 base64 characters), and the wrapping `data:` URI adds a MIME-type/`;base64,` prefix on top — confirmed by Terence Eden's writeup ([shkspr.mobi](https://shkspr.mobi/blog/2021/08/you-dont-need-external-assets-in-an-html-file/)) and CSS-Tricks' data URI reference ([css-tricks.com/data-uris](https://css-tricks.com/data-uris/)). CSS-Tricks also documents a hard historical ceiling — IE8 capped data URIs at 32,768 bytes — and gives the practical framing that a 300KB CSS file with embedded images can still be a net win over a 50KB file plus 6 external HTTP requests **only if** the container is cached; for a jimemo output that is viewed once and not cached, that tradeoff doesn't apply the same way, so assets should stay small in absolute terms rather than relying on a request-savings argument.
- **Data URI vs inline SVG.** For vector icons/illustrations, inlining raw `<svg>` markup directly in the HTML avoids the base64 33% tax entirely and lets CSS/JS touch the SVG's DOM (recolor via `currentColor`, animate paths); CSS-Tricks recommends `data:` URIs mainly for raster formats or when the SVG must live inside a CSS `url()` (e.g., a `background-image`) where raw markup isn't accepted ([css-tricks.com/data-uris](https://css-tricks.com/data-uris/)).
- **Real-world performance caveat.** Catchpoint's field measurement across mobile sessions (a ~2013-era study run on then-current iOS 6 / Android 2.x hardware, not a current benchmark) found images loaded via data URI averaged 6x slower than the same image loaded via a normal `src` link ([catchpoint.com/blog/data-uri](https://www.catchpoint.com/blog/data-uri)), because the browser can no longer parallelize/cache the asset independently of the HTML document. The underlying mechanism (no independent caching/parallelization for inlined assets) still holds architecturally, but the magnitude is dated and should not be quoted as current. For jimemo this argues for keeping embedded raster images few and small, and preferring inline SVG/CSS for anything that can be vector or code.
- **`prefers-color-scheme` + manual override pattern.** The robust pattern documented across MDN and independent write-ups is a three-state model — `system` / `light` / `dark` — stored in a single attribute on the root element (e.g. `data-theme`), with CSS custom properties set in `:root`, overridden by `:root[data-theme="dark"]`/`:root[data-theme="light"]` for explicit choices, and by `@media (prefers-color-scheme: dark) { :root[data-theme="system"] { ... } }` for the default/system case ([MDN prefers-color-scheme](https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-color-scheme); pattern write-up at [cr0x.net](https://cr0x.net/en/dark-mode-toggle-pattern/)). A two-state (light/dark only) toggle is explicitly called out as inferior because it forces a guess about what "off" means and doesn't degrade gracefully when JS is disabled — relevant to jimemo since a static single-file HTML page may be opened with scripting disabled.
- **Print stylesheet essentials.** Core techniques used across guides: a `@media print` block (or dedicated stylesheet) that hides non-content chrome (`display: none` on navigation/controls), `page-break-inside: avoid` on tables/figures/code blocks so they don't split mid-element, `page-break-before`/`page-break-after` to force clean section starts, and the `@page` at-rule to control margins/size ([overview at customjs.space print CSS cheatsheet](https://www.customjs.space/blog/print-css-cheatsheet/); page-break reference at [css-tricks.com/almanac/properties/p/page-break](https://css-tricks.com/almanac/properties/p/page-break/)). For jimemo, print rules matter even for a browser-only artifact because "save as PDF" is the most likely conversion path a user takes with a single HTML file.
- **Font strategy: system stack vs embedded WOFF2.** A modern subsetted WOFF2 weight typically runs 15–40KB thanks to Brotli compression (WOFF2 gets ~60–70% smaller than the source font, itself ~30% smaller than WOFF); an unoptimized web font set can be reduced from ~672KB to ~56KB with subsetting and modern formats ([web font optimization guide](https://font-converters.com/guides/web-font-optimization)). Against that, a system font stack (e.g. `-apple-system, Segoe UI, Roboto, ...`) costs zero bytes and zero network/render latency, at the expense of visual consistency across OSes. The commonly cited compromise is to spend the embedding budget only on display/heading type and rely on system fonts for body text where legibility and load speed matter most ([etienneaubertbonn.com system vs web fonts](https://www.etienneaubertbonn.com/system-fonts-vs-web-fonts/)). For a strict single-file, no-remote-fetch constraint, this means: default to a system font stack, and treat embedded WOFF2 as an explicit, size-budgeted opt-in per theme rather than a default.

## Ideas worth stealing (credited)

- **Sidenotes and margin notes for report-style documents** — Dave Liepmann / Edward Tufte project, [Tufte CSS](https://github.com/edwardtufte/tufte-css), MIT license. Tufte CSS puts footnote-style annotations in the page margin next to the referenced text instead of at the page bottom, with a CSS-only (no JS) toggle for small screens contributed by Chris MacKay. This is directly applicable to jimemo's "report" template: citations, asides, and data provenance notes could live in a margin column rather than breaking reading flow.
- **Zero-class, semantic-HTML-first theming** — picocss org, [Pico CSS](https://github.com/picocss/pico), MIT license. Pico ships automatic light/dark themes and ~20 handcrafted color palettes under 10KB compressed, targeting plain semantic tags (`header`, `main`, `article`) with no authored classes required. jimemo's default theme could adopt this "classless first, escape hatch classes second" philosophy so generated templates stay readable and hand-editable.
- **Explicit warnings over silent failure when an asset can't be inlined** — pytest-dev, [pytest-html user guide](https://github.com/pytest-dev/pytest-html/blob/master/docs/user_guide.rst), MPL-2.0 license. Rather than silently leaving a linked file as an external reference in "self-contained" mode, pytest-html surfaces a warning. jimemo's build/render step should follow this precedent: any asset that can't be embedded (too large, unreadable, remote URL) should hard-fail or loudly warn rather than quietly producing a page with a broken remote dependency.
- **Three-state (system/light/dark) theme attribute pattern, not a two-state toggle** — independent blog post by the author of [cr0x.net](https://cr0x.net/en/dark-mode-toggle-pattern/), "Dark mode that doesn't suck: prefers-color-scheme + manual toggle pattern". The post argues a two-state light/dark toggle is a design smell because it forces a guess about what "off" means and gets stuck once the OS preference changes; the fix is a single `data-theme="system|light|dark"` attribute on the root element with CSS reacting to both the attribute and `prefers-color-scheme`. jimemo's theming should adopt this three-state model rather than a simpler binary toggle.
